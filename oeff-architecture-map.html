05. Data Architecture

Not a Tree — A Hub with Rings

Kim's insight: "The most specific our data will ever get is by event." But the data isn't hierarchical — Directory links to Events, Venues, Films, Assets, and Deliverables. Assets link to Films AND Events AND Venues. Deliverables draw from multiple Assets. This is a directed graph with Events at the center, not a tree with a single root.

MASTER TABLES (persistent, year-over-year)
  Events          The relational hub — links venue + film + contacts + date/time
      Screenings     primary event type
      Action Fairs   community events (future subtype)
      Panels, Concerts   additional subtypes as needed
  Directory       All people — one record per person, typed by role + priority
      Replaces Host Contacts + Film Contacts (~36 records merged)
      Roles: Host, Venue Contact, Filmmaker, Distributor, Panelist, Volunteer Lead, AV Tech, Partner Org
      Links to Events, Venues, Films (one person can hold multiple roles)
  Venues          Physical spaces — capacity, ADA, AV, address
      + Classification (Flagship / Community)
      + Technical Tier (T1–T4)
      + Linked Directory (replaces old Host Contacts link)
  Films           Titles, runtimes, licensing terms
      + Outreach Status, File Status, Delivery Status
      + Runtime (minutes) — currently missing, blocks run-of-show
      + Social Handles, Premiere Status
  Sponsors        Organizations, tiers, financial commitments
  Members         14k email list (conceptual — served by Mailchimp/Mailmeteor, deferred as table)

RELATIONAL + TRACKING (season-specific, cross-linked to master tables)
  Host Helper 2026      28 flat rows, ~28 text/date/URL fields — script-assembled, team-editable
      One row per event → one Interface Designer page per event
      21 fields script-owned (overwritten on re-run)
      7 fields team-editable (preserved: setup notes, AV contact, custom instructions, etc.)
  Assets                Master index of all inbound/outbound/internal files
      Categories: Inbound Film, Inbound Press, Outbound Host, Outbound Filmmaker, Internal Ops, Marketing
      Types: Screening Copy, Trailer, SRT, Poster, Headshot, Slide Deck, Guide, Packet, Template
      Links ↔ Films, Events, Venues, Directory
  Asset Versions        QC history per file version — preserves version trail
      QC Status: Pending, Pass, Fail, Conditional
      Self-link (Supersedes) for version chains
  Deliverables          Packaged outputs for stakeholders ↔ Events, Assets (many:many)
      Types: Screening Packet, Presenter Guide, Slide Deck, Marketing Kit, Webinar Link, Screening Report
      Audience: Host, Filmmaker, Internal, Partner, Press
  Delivery Log          Recipient-level tracking — who got what, when ↔ Deliverables, Directory
  Comms Log             Outreach tracking ↔ Events, Directory
  Host Confirmations    Host response tracking — links to Events (currently not linked to Venues)
      3 automations built (verify: are they actually running?)
  Host Intake           Form responses from venue intake process
  Event Media Assets    Posters, stills, press photos ↔ Events
  Packet QA             Quality checks on screening packets ↔ Events, Assets
  Webinar Content       Recorded sessions, slide decks for host training
  Participants          Event attendees / participants (operational tracking)
  Recordings            Session recordings linked to events
  Merged Timeline       Cross-event calendar view for scheduling conflicts

STAKEHOLDER OUTPUTS (script-assembled, design-forward resources for external audiences)
  Host Helper Pages     Per-event flat views for host venues (Interface Designer)
      Data source: Host Helper 2026 table → one page per event
      Audience: venue contacts, host coordinators
  Screening Packets     Bundled deliverables for day-of operations
      Contents: film copy, SRT, presenter guide, slide deck, run-of-show
      Assembled from Assets + Deliverables tables
  Filmmaker Kits        License docs, screening reports, press assets for filmmakers
      Post-festival: audience data, photos, host feedback
  Email Campaigns       Mailmeteor merge data for host/filmmaker outreach
      Draws from Directory + Events + Comms Log

RESILIENCE (operational infrastructure — makes the system survivable)
  Routing Rules         Encodes "where does this go?" in data, not in someone's head
      Replaces "ask Garen" as the routing table
  Integration Registry  Every script/sync: owner, schedule, failure mode, last successful run
      Source table, destination system, run cadence
      Enables contractor handoff: "this field is authoritative in X only"
  Seasons               Year/edition record for cross-season reporting (2027+)
      Prevents copied-base drift; enables year-over-year analysis
  Decision Log          Architecture decisions captured in ~/decisions/ (git-backed)
      Linked from Routing Rules for rationale trail
Why rings, not a tree: A tree implies single-parent containment. OEFF's data has many-to-many cross-links — Directory connects to Events, Venues, Films, Assets, and Deliverables simultaneously. Assets link to Films AND Events AND Venues. The concentric ring model shows proximity to the core (Events) without forcing a false hierarchy. Data flows outward: core tables feed the relational/tracking layer, which feeds stakeholder-facing outputs (host pages, screening packets, filmmaker kits, email campaigns). Nothing flows back upstream.
06. Architecture Comparison

v1 (Venue-Centric) vs v2 (Event-Centric)

The March 13 meeting surfaced why the venue-centric model broke: multi-event venues produced concatenated values, rollup chains were uneditable, and formula fields obscured what was actually stored.

v1 — Venue-Centric

  • Organizing principle: Venues
  • Host Helper: Per-venue shared grid views (24)
  • Data chain: 3-layer rollup (link → LKP → Host formula)
  • Kim's edit model: Cannot edit rollups or formulas
  • Multi-event venues: Concatenated values with commas
  • Contacts: Separate Host + Film Contact tables
  • Debuggability: Trace link → rollup → formula
  • Venues table: 69 fields (29 rollup/formula debt)

v2 — Event-Centric

  • Organizing principle: Events
  • Host Helper: Per-event Interface Designer pages
  • Data chain: Flat text fields, script-assembled
  • Edit model: Team edits flat fields directly
  • Multi-event venues: One row per event — no ambiguity
  • Contacts: Unified Directory with role + priority
  • Debuggability: What you see is what's stored
  • Host Helper: 28 fields, all text/date/URL
Core tradeoff: v2 duplicates data (every event row has its own copy of the venue address) in exchange for editability and clarity. The assemble-host-helper.py script can re-flatten at any time without breaking Kim's manual edits in the 7 protected fields.
07. Architecture Decisions

7 Decisions from March 13 Meeting

Kim and Garen's 63-minute call produced 7 foundational decisions. These reshape how every downstream tool — Host Helper, sync scripts, merge sheets — relates to the data.

Decision 1

Events as relational key

Events/screenings are the most specific data level. They link to master tables (venues, films, contacts) rather than the reverse.

"The most specific our data will ever get is by event." — Kim
Decision 2

Unified Directory

All people-data in one table with a Contact Type field, replacing scattered host/venue/filmmaker contact lists.

"What if we just called this table a directory?" — Kim
Decision 3

Partnership categorization

Collaborators (mission-oriented) vs. transactional contacts (business relationship where money changes hands for a deliverable).

Decision 4

Members vs Partners

14k email list = members (who we work for). Active stakeholders = partners (who we work with). Different interaction levels, different data needs.

"Distinguish who we work with and who we work for." — Kim
Decision 5

Per-event interface pages

Replace 24 shared grid views with event-centric Interface Designer pages. Record-parameter deep links give Kim direct edit access per event.

Decision 6

Dual venue classification

Flagship/Community (team presence level) alongside T1–T4 (technical needs). Two separate systems, not competing.

Decision 7

"Events" umbrella term

"Screenings" becomes a subtype of "events." Events covers action fairs, concerts, panels. Future-proofs the data model.

09. Data Flow

Pipeline Architecture

Three canonical sources feed the system. Kim's tracker sheets carry process knowledge and validated operational data. The OEC Active Roadmap carries team-facing planning decisions. Airtable holds relational structure. Scripts bridge all three.

Kim's Tracker (5 tabs) → validated contacts, venues, AV, films → Directory + Venues + Films
OEC Active Roadmap → scheduling, film assignments, confirmations → Events table
Kim's Comms Tracker (2 tabs) → outreach history, RACI, engagement log → Comms Log + Deliverables
Events table links to all master tables
Events + Venues + Films + Contacts → assemble-host-helper.py → Host Helper 2026
Host Helper 2026 → Interface Designer → Host-facing pages
Host Helper 2026 → oeff-airtable-sync.py → Mailmeteor campaigns
Key insight: Data flows downhill. Master tables feed Events. Events feed the assembly script. The script produces flat rows. Interface Designer presents them. Merge sheets consume them for email campaigns. No data flows upstream from assembled tables to master tables.

15. Anti-Patterns

10 Things to Avoid

From Codex research: failure modes seen across festival and nonprofit data architectures. Each annotated with OEFF's current exposure and mitigation.

1. Copy-and-edit workflows

Risk: Copying tables/sheets for editing creates drift. Mitigation: Host Helper is assembled, not copied. Script re-flattens from canonical sources.

2. Assembled tables treated as canonical

Risk: Edits in output tables never flow back to master. Mitigation: Clear separation — canonical edits go in core tables, assembled layer is regenerated.

3. Role explosion via separate people tables

Risk: Host Contacts, Film Contacts, Panelists, Sponsors all separate. Mitigation: Unified Directory with role/context fields.

4. Stuffing role + priority into one field

Risk: Semantic collision ("Primary Filmmaker" vs "Host who is Primary"). Mitigation: Directory has separate Role, Priority, and Relationship Type fields.

5. Routing via memory only

Risk: "Ask Garen" is the current routing table. Mitigation: Routing Rules table. Decision log in ~/decisions/ captures rationale now.

6. Silent bidirectional edits

Risk: Two systems editing the same field without ownership rules. Mitigation: Script-owned vs team-editable split. Sync scripts have explicit direction.

7. No version records for assets

Risk: Overwriting links with no history. Mitigation: Asset Versions table in blueprint. Current: file tracker script logs versions.

8. Text filter instead of formal Season record

Risk: Year-over-year ambiguity. Mitigation: Currently using "2026 Events" link field. Formal Seasons table is a 2027 target.

9. Over-normalizing too early

Risk: Many tiny tables with sparse usage. Mitigation: Add tables only for real many-to-many or independent lifecycles. Every new table must answer: "what lifecycle can't a field represent?"

10. Permission shortcuts for interns

Risk: Broad base editor access for convenience. Mitigation: Interface-only access, narrow forms, field constraints via Airtable permissions.

19. Scale Transitions

When to Outgrow Airtable + Scripts

At OEFF's current size (~30 events, ~16 films, small seasonal team), Airtable + scripts is the right core. Transition pressure appears when workforce complexity exceeds what one Airtable base can support — not when data grows.

TriggerToolWhen
Submission volume outgrows manual intake mappingFilmFreeway (inbound) + Airtable (ops)If submissions exceed ~100/year
Hybrid/virtual rights windows need integrated deliveryFestival Scope + Shift72If virtual screenings become significant
Team grows beyond ~15 with department silosPurpose-built platform (Eventival)If org structure formalizes
Key insight from research: Workforce complexity triggers the tool transition earlier than data size in contractor-heavy teams. OEFF's current pragmatic tool split: Google Forms for inbound submissions, Airtable for ops, stdlib Python for integration. This is the right stack for 2026.
