02. Data Hierarchy

Three-Layer Architecture

Kim's insight: "The most specific our data will ever get is by event." Master tables are persistent and year-over-year. The relational layer is season-specific. The assembled layer is script-generated and Kim-editable.

MASTER TABLES (persistent, year-over-year)
  Directory     All people — one record per person, typed by role + priority
  Venues        Physical spaces — capacity, ADA, AV, address
  Films         Titles, runtimes, licensing terms
  Members       14k email list (separate from active partners)

RELATIONAL LAYER (season-specific)
  Events        Links a venue + film + contacts + date/time
    Screenings  (primary event type)
    Action Fairs, Panels, Concerts (future subtypes)

ASSEMBLED LAYER (script-generated, Kim-editable)
  Host Helper 2026  28 flat rows with ~28 text/date/URL fields
    One row per event → one Interface Designer page per event
Why three layers: Master tables survive across festival years. The relational layer (Events) is rebuilt each season by linking master records. The assembled layer flattens everything into human-readable rows that Kim can edit directly — no formulas, no rollups, no link chains to trace.
03. Architecture Comparison

v1 (Venue-Centric) vs v2 (Event-Centric)

The March 13 meeting surfaced why the venue-centric model broke: multi-event venues produced concatenated values, rollup chains were uneditable, and formula fields obscured what was actually stored.

v1 — Venue-Centric

  • Organizing principle: Venues
  • Host Helper: Per-venue shared grid views (24)
  • Data chain: 3-layer rollup (link → LKP → Host formula)
  • Kim's edit model: Cannot edit rollups or formulas
  • Multi-event venues: Concatenated values with commas
  • Contacts: Separate Host + Film Contact tables
  • Debuggability: Trace link → rollup → formula
  • Venues table: 69 fields (29 rollup/formula debt)

v2 — Event-Centric

  • Organizing principle: Events
  • Host Helper: Per-event Interface Designer pages
  • Data chain: Flat text fields, script-assembled
  • Kim's edit model: Edits flat fields directly
  • Multi-event venues: One row per event — no ambiguity
  • Contacts: Unified Directory with role + priority
  • Debuggability: What you see is what's stored
  • Host Helper: 28 fields, all text/date/URL
Core tradeoff: v2 duplicates data (every event row has its own copy of the venue address) in exchange for editability and clarity. The assemble-host-helper.py script can re-flatten at any time without breaking Kim's manual edits in the 8 protected fields.
04. Architecture Decisions

7 Decisions from March 13 Meeting

Kim and Garen's 63-minute call produced 7 foundational decisions. These reshape how every downstream tool — Host Helper, sync scripts, merge sheets — relates to the data.

Decision 1

Events as relational key

Events/screenings are the most specific data level. They link to master tables (venues, films, contacts) rather than the reverse.

"The most specific our data will ever get is by event." — Kim
Decision 2

Unified Directory

All people-data in one table with a Contact Type field, replacing scattered host/venue/filmmaker contact lists.

"What if we just called this table a directory?" — Kim
Decision 3

Partnership categorization

Collaborators (mission-oriented) vs. transactional contacts (business relationship where money changes hands for a deliverable).

Decision 4

Members vs Partners

14k email list = members (who we work for). Active stakeholders = partners (who we work with). Different interaction levels, different data needs.

"Distinguish who we work with and who we work for." — Kim
Decision 5

Per-event interface pages

Replace 24 shared grid views with event-centric Interface Designer pages. Record-parameter deep links give Kim direct edit access per event.

Decision 6

Dual venue classification

Flagship/Community (team presence level) alongside T1–T4 (technical needs). Two separate systems, not competing.

Decision 7

"Events" umbrella term

"Screenings" becomes a subtype of "events." Events covers action fairs, concerts, panels. Future-proofs the data model.

06. Data Flow

Pipeline Architecture

Dual-canonical principle: Kim's Google Sheets are canonical for validated data (she checks it). Airtable is canonical for relational data (links between tables). Sync scripts bridge them.

Kim's Tracker Sheet → one-time import → Directory + Venues + Films
Events table links to all master tables
Events + Venues + Films + Contacts → assemble-host-helper.py → Host Helper 2026
Host Helper 2026 → Interface Designer → Host-facing pages
Host Helper 2026 → oeff-airtable-sync.py → Mailmeteor campaigns
Key insight: Data flows downhill. Master tables feed Events. Events feed the assembly script. The script produces flat rows. Interface Designer presents them. Merge sheets consume them for email campaigns. No data flows upstream from assembled tables to master tables.

09. Unified Directory

Directory Table Design

Replaces Host Contacts (52 records) + Film Contacts (14 records) with one table. ~66 records with deduplication. A person who is both a host contact and a panelist gets one record with multiple roles.

Directory (replaces Host Contacts + Film Contacts)

  FIELDS
  Name             singleLineText (primary)
  Email            email
  Phone            phoneNumber
  Organization     singleLineText
  Role             singleSelect: Host, Venue Contact, Filmmaker, Distributor,
                     Panelist, Volunteer Lead, AV Tech, Partner Org, Staff
  Priority         singleSelect: Primary, Secondary, Marketing, AV/Tech, Intern
  Relationship Type singleSelect: Collaborator, Transactional, Internal
  Notes            longText

  MIGRATION PATH
  Host Contacts    → Directory (Role=Host or Venue Contact)
  Film Contacts    → Directory (Role=Filmmaker or Distributor)
  Kim's Tracker    → Directory (validated contact list import)

10. Industry Patterns

8 Patterns from Codex Research

Cross-referenced against TIFF, Tribeca, Eventival/Eventive, Salesforce NPSP, HubSpot, VolunteerHub, Better Impact, and Eventbrite. Each pattern maps to OEFF's current or target architecture.

Pattern 1

Canonical People Table + Role Layer

Salesforce NPSP, HubSpot, Bloomerang. One person record with relationship layers, not one table per function.

OEFF: Directory table
Pattern 2

Intake as Inbound Boundary

TIFF (FilmFreeway), Tribeca (dual intake). Submission systems for intake, separate ops elsewhere.

OEFF: Host Intake → fold into Venues
Pattern 3

Season/Edition as First-Class Entity

Eventival, FilmFreeway. Edition-based sync, season archiving.

OEFF: 2027 hardening target
Pattern 4

Event-Centric Core

Eventive, SXSW, CES, Pitchfork. Events as atomic ops unit; sessions as optional children.

OEFF: v2 architecture (active)
Pattern 5

Asset Registry + Version Rows

FilmFreeway, Eventival. Separate "asset exists" from "asset was delivered to X."

OEFF: Codex recommends 5 new tables
Pattern 6

Permission by Surface

Airtable interface-only users, volunteer platform visibility rules. Access via views/interfaces, not cloned data.

OEFF: Interface Designer pages
Pattern 7

Single Source + Sync Contracts

Airtable sync docs, Eventival overwrite semantics. Source overwrites destination; ownership is explicit.

OEFF: Script ownership model
Pattern 8

Turnover Resilience as Design Requirement

Sundance (Bloomerang + Litmos), VolunteerHub, Better Impact. Onboarding is procedural, not oral tradition.

OEFF: Routing Rules + Data Dictionary
What the research validates: OEFF's v2 architecture aligns with 6 of 8 patterns already. Pattern 3 (Seasons) and Pattern 5 (Asset Registry) are 2027 hardening targets. At ~30 events and a small seasonal team, Airtable + stdlib Python scripts remains the right core. Transition pressure comes from submission volume or multi-department workflow complexity, not data size.
11. Anti-Patterns

What to Avoid

From the Codex research: failure modes seen across festival and nonprofit data architectures. Each is annotated with OEFF's current exposure.

Copy-and-edit workflows

Risk: Copying tables/sheets for editing creates drift. Mitigation: Host Helper is assembled, not copied. Script re-flattens from canonical sources.

Stuffing role + priority into one field

Risk: Semantic collision ("Primary Filmmaker" vs "Host who is Primary"). Mitigation: Directory has separate Role, Priority, and Relationship Type fields.

Routing via memory only

Risk: "Ask Garen" is the current routing table. Mitigation: Routing Rules table (2027 target). Decision log in ~/decisions/ captures rationale now.

Silent bidirectional edits

Risk: Two systems editing the same field without ownership rules. Mitigation: Script-owned vs Kim-editable split. Sync scripts have explicit direction (Airtable → Sheet, not both ways for the same field).

No version records for assets

Risk: Overwriting links with no history. Mitigation: Asset Versions table in Codex blueprint (post-festival). Current: file tracker script logs versions.

Text filter instead of formal Season record

Risk: Year-over-year ambiguity. Mitigation: Currently using "2026 Events" link field. Formal Seasons table is a 2027 target.

14. Handoff Resilience

What Survives Without Garen or Kim

From Codex assessment: OEFF's main risk is continuity, not database scale. The architecture should be self-documenting enough that a new contractor can operate it within a day.

If Garen Leaves

  • At risk: Script refresh, sync triage, Cloudflare deployment
  • Strong: Event-centric model is structurally clear
  • Tribal knowledge: Why fields are canonical vs assembled; rerun protocol; failure handling
  • Mitigation: MANIFEST.md, decision log, this architecture map

If Kim Leaves

  • At risk: Daily data hygiene, comms-state updates
  • Strong: Flat tables reduce formula-debug burden
  • Tribal knowledge: Interface conventions, update rhythms
  • Mitigation: Playbooks, field descriptions, per-event pages
Minimum self-documentation (from Codex): Data Dictionary table, Table Purpose sheet, Script Registry, Runbook docs for top 10 workflows, Decision Log, Season Closeout checklist. Of these, the Decision Log exists (~/decisions/), the Script Registry exists (MANIFEST.md), and the architecture map (this document) covers Table Purpose.
